## Executive residual gaps (audit §1 items 1–10)

These are the open architectural seams named in the executive verdict. Each maps to a P0 below.

| # | Residual gap (verbatim intent) | Tracks |
|---|--------------------------------|--------|
| 1 | Cluster mode starts `NodeRuntime`, but ordinary public SQL/Kit/native data requests still execute against standalone `AppState.db` | P0.2 |
| 2 | Tablet storage uses one hidden opaque key/value table rather than real user table schemas, columns, constraints, RLS, and index generations | P0.3 |
| 3 | Distributed SQL still uses `LogicalPlanLite`; full DataFusion lowering is out of scope in current source | P0.4 |
| 4 | HLC exists as replicated metadata, but row visibility and snapshots are still fundamentally epoch-based | P0.5 |
| 5 | Public scoped-service-principal handle lets an in-process caller supply its own permission list | P0.1 |
| 6 | Native query status and cancellation do not expose the complete durable-outcome contract required for safe write recovery | P0.6 |
| 7 | Generated embeddings lack a cryptographic semantic-model fingerprint and a public query-text embedding/re-embedding generation workflow | P0.7 |
| 8 | Distributed AI performs tablet-local fusion and then one global RRF contribution per row; it does not globally fuse dense, sparse, MinHash, and lexical contributions correctly | P0.8 |
| 9 | Fragment and AI workers exist but are not installed by the production server startup path | P0.2 / P0.4 |
| 10 | Release qualification validates R1–R10 from the previous audit, not every Stage 0–5 acceptance gate from the full architecture specification | P0.9 |

**Product claims not closed (audit §1 overall decision):** production HA through public client traffic; production sharded SQL; production distributed AI retrieval; full architecture “Qualified”.

# Definition of done (audit §23)

The complete architecture is done only when **all** of the following are simultaneously true:

1. Desktop applications use safe embedded exclusive or authenticated shared-handle access.
2. The single-node server remains fully supported.
3. Cluster mode routes ordinary public data traffic through consensus and real tablet storage.
4. Tablets store typed MongrelDB user tables with the same MVCC, constraints, security, and indexes as standalone mode.
5. Real DataFusion SQL plans execute across tablet processes.
6. One HLC timestamp defines row visibility everywhere.
7. Generated and query embeddings are tied to immutable semantic model fingerprints.
8. Distributed hybrid retrieval globally fuses named retrievers and performs bounded exact reranking.
9. Every work class is governed by one node admission and memory system.
10. Native and HTTP clients preserve identical durable write outcomes.
11. Every mandatory Stage 0–5 acceptance gate passes for one exact release SHA and packaged artifacts.

Only then may MongrelDB truthfully claim:

```text
embedded desktop/local database
production single-node website database
replicated high-availability database
sharded distributed SQL database
workload-isolated distributed AI retrieval platform
multi-tenant enterprise database
```
